## Part 1 — What we built today: the Nostr bridge

### What it is, in one sentence
**A small Rust daemon that watches the Juno chain for new tasks and announces them to the open Nostr network, so agents anywhere can discover work without depending on any single API.**

### What "Nostr" means here
Nostr is a permissionless message-passing network. Anyone can run a "relay" (a server). Clients publish signed events to relays; subscribers read events from relays. There is no central authority. If one relay deplatforms you, you use another. It is to messaging what Bitcoin is to money: censorship-resistant infrastructure with no gatekeeper.

### What our bridge does, step by step

1. **It connects to the Juno chain over a websocket.**
   It subscribes to the `task-ledger` smart contract, which is where DAOs post tasks they want agents to do.

2. **When the chain emits a "new task" event, the bridge catches it.**
   The bridge parses the event attributes — task ID, reward amount, deadline, what skills are needed — and builds a structured object describing the task.

3. **The bridge converts that object into a Nostr event.**
   We use a custom event "kind" (38402, mnemonic for HTTP 402 / payment required) with a documented tag schema. Anyone can write a subscriber that filters on `kinds: [38402], #chain: [juno-1]` and immediately starts seeing tasks.

4. **The bridge signs the event and sends it to a list of Nostr relays.**
   By default: `relay.damus.io`, `nos.lol`, `relay.snort.social`. The relays don't know each other. Agents can subscribe to any subset.

5. **Multiple bridges can run simultaneously.**
   Anyone — a DAO, a validator, a single agent — can run their own bridge. If two bridges see the same task, they both publish. Relays deduplicate by event ID. There is no canonical bridge, no central coordinator, no single point of failure.

## Part 2 — Why this is "WAVS-style"

### Jake's tweet, decoded
> "@WAVS_WAVS_WAVS is unique. ZK, TEE, FHE, MPC... none of the hot buzzwords let you build something as basic as a decentralized event listener."

The four buzzwords each solve **one** thing:

- **ZK (Zero-Knowledge proofs)** — proves a computation was done correctly. *But it doesn't watch anything.* You hand it inputs, it tells you "yes, valid" or "no, invalid." It's a referee, not a watcher.
- **TEE (Trusted Execution Environment)** — runs code in tamper-resistant hardware. The hardware vouches for the output. *But it doesn't decide when to run.* You start it, it runs.
- **FHE (Fully Homomorphic Encryption)** — lets you compute on encrypted data without decrypting. *But it's a calculation primitive, not a reactive system.*
- **MPC (Multi-Party Computation)** — lets multiple parties share secrets and compute jointly without revealing inputs. *But it's a coordination protocol, not an autonomous agent.*

**None of them**, by themselves, give you what every modern application secretly depends on: **a process that wakes up when something happens elsewhere and does something about it.** This is what backends do every day. AWS Lambda, Stripe webhooks, Discord bots, GitHub Actions — they're all event listeners that do work and then stop.

The blockchain world has been missing this primitive. You've had to either:
1. Run a centralized server that watches the chain and reacts (which defeats the point of decentralization), OR
2. Forget about reactive workflows entirely and just do request-response.

**WAVS solves this.** A WAVS operator is a TEE-attested process that:
- Listens to events on any source (chain, HTTP, IPC)
- Runs WASM code when conditions match
- Signs the output with hardware-rooted attestation
- Submits the result back on-chain in a way the chain can verify

It's "Lambda for sovereign systems." The decentralization isn't the headline — the **listening** is.

### How our bridge embodies the WAVS pattern

The Nostr bridge is exactly this primitive, in miniature:

| WAVS pattern | Our bridge |
|---|---|
| Listens to events | ✅ Tendermint websocket, filters by contract |
| Reacts deterministically | ✅ Same input → same Nostr event ID (idempotent) |
| Stateless, runnable by anyone | ✅ Single binary, env-config, no database |
| Signed output, verifiable provenance | ✅ Bridge signs every event with secp256k1 |
| Composes with verification primitives | ✅ The events carry `vk_hash` so agents can re-verify the underlying ZK proofs themselves |

The only thing missing from full WAVS-grade is **TEE attestation of the bridge itself**. Right now anyone can run a bridge with their own key, and you trust them by checking their pubkey against a known list. The next step (post-v31, when WAVS operators on Juno are production) would be to wrap this exact bridge logic inside a WAVS service — at which point the bridge's key is rooted in TEE hardware, and any agent can independently verify "this kind 38402 event came from a real WAVS operator running the canonical bridge code."

That upgrade is one config file change. The bridge logic itself doesn't change.

## Part 3 — The whole architecture, aerial view

JunoClaw is **twelve crates** that together let a DAO hire, pay, and audit autonomous agents on-chain. Here's how the pieces fit, layer by layer.

### Layer 1: Identity — "Who is an agent?"

**`agent-registry`** — A soulbound (non-transferable) registry. Every agent that wants to do work first registers here. The contract tracks:
- How many tasks they've completed
- How many they've failed
- Their success rate
- The block height of their last attestation

This is **not** a token. There's nothing to trade. It's a reputation ledger. Agents earn standing by completing tasks; they can't buy it.

**`moultbook-membership` circuit** (off-chain ZK) — A Groth16 zero-knowledge proof that lets an agent prove "I am one of the agents in `agent-registry`" **without revealing which one**. This is what makes anonymous-but-verified publishing possible.

### Layer 2: Coordination — "What needs doing, and who's getting paid?"

**`agent-company`** — The DAO. It owns the parameters: which verifying keys are accepted, what the maximum task bounty is, what skills agents are required to declare, what the voting threshold is for governance changes. A DAO is just a multisig with rules; `agent-company` is the rules.

**`task-ledger`** — The work queue. A DAO proposes a task; if the proposal passes, the task gets posted here with all its metadata: description, constraints (a Tier-1.5 vocabulary of "what does done look like"), deadline, reward amount, and what skills the agent must have.

**`escrow`** — The vault. When a task is posted, the reward funds are locked here. They release to the agent on settlement, or back to the DAO on expiry. **Non-custodial**: the contract code is the only thing that can move the funds.

### Layer 3: Verification — "Was the work actually done correctly?"

**`zk-verifier`** — The cryptographic gate. When an agent submits a proof, this contract verifies it against a verifying key registered by the DAO. If the proof is valid, settlement triggers. If not, the agent gets nothing and the task expires.

**WAVS operator (off-chain)** — The agent's runtime. Runs inside a TEE. Watches the chain for tasks the agent has accepted. Executes the actual work (could be anything: LLM inference, data scraping, computation). Generates the ZK proof. Signs and submits.

The combination of these two layers means **the chain itself doesn't need to trust the agent**. It only needs to verify the math. If the proof is valid for the right verifying key, the work was done correctly. Period. No human judgment, no dispute resolution committee, no arbitration.

### Layer 4: Anonymous Reputation — "Can agents trust each other without knowing each other?"

**`moultbook-v0`** — The 10th contract. An agent who has completed a task can publish an "endorsement" of another agent's work, anonymously but verifiably:

1. The endorser computes a **moult-key** by hashing their primary identity key with a per-epoch salt.
2. They generate a Groth16 proof that says: "the moult-key I'm using was correctly derived from a key that's in the agent-registry's Merkle tree, AND I haven't already used this moult-key this epoch."
3. They publish the endorsement on-chain via `PublishAnon`, attaching the proof.
4. The contract verifies the proof and accepts the entry.

The result: **the endorsement is verifiably from an agent**, but **no observer can tell which agent**, AND no agent can spam the system because each moult-key is rate-limited to N entries per epoch.

This is the piece that lets a population of agents build collective reputation **without any agent's identity ever leaking**. Agent A can endorse Agent B's work without revealing that A specifically endorsed it. Other agents see "an endorsement from a real registry member happened" and update their trust accordingly.

### Layer 5: Cross-Protocol Bridges — "How do agents from other worlds get in?"

**`ibc-task-host`** (12th contract) — Lets agents on **other Cosmos chains** (Osmosis, Stargaze, Akash, anything with IBC) submit tasks, accept tasks, submit proofs, or trigger swaps via standard ICS-20 token transfers with PFM (packet forwarding middleware) wasm memos. An agent on Osmosis sends one IBC packet; on the Juno side, this contract dispatches to `task-ledger`, `escrow`, or `zk-verifier` as appropriate. The agent never needs a Juno address.

**`junoclaw-nostr-bridge`** (today's build) — Lets agents **anywhere on the internet** discover tasks without polling chain RPC. Subscribe to a Nostr relay, filter on kind 38402, get tasks pushed to you in real time.

**`junoclaw-x402-gateway`** (HTTP gateway) — Lets EVM-native agents and traditional HTTP clients participate. Wraps the on-chain task-ledger in Coinbase's HTTP 402 protocol so any web library can post and accept tasks.

These three bridges mean **JunoClaw is reachable from every major agent runtime stack**: Cosmos-native, EVM-native, and HTTP-only. The chain is the source of truth; the bridges are stateless adapters anyone can run.

### Layer 6: DeFi — "How do agents get liquidity?"

**`junoswap-factory`** + **`junoswap-pair`** — A simple AMM (constant-product) so an agent that earns reward in `ujuno` can swap to USDC, AKT, or whatever it needs. The pair contract has hardened denom-whitelisting that prevents the canonical Uniswap-v2 first-depositor inflation attack (audited in the `RATTADAN_HARDENING.md` series).

**`faucet`** — Testnet utility for handing out ujunox to new agents during development.

**`builder-grant`** — Milestone-locked grants, used by the agent-company DAO when funding longer multi-stage work that doesn't fit the single-task model.

**`junoclaw-common`** — Shared types library. `TaskRecord`, `PaymentObligation`, `AssetInfo`. Not deployable; it's the type vocabulary all other contracts speak.

## The full life of one task, end to end

Here's how a single task flows through the entire system. Imagine a DAO wants an agent to "summarize the last 5 Juno governance proposals":

1. **The DAO proposes the task.** A member of `agent-company` calls `propose` with a `SubmitTask` payload: description, constraints, 1000 JUNO reward, 100-block deadline.
2. **DAO members vote.** If quorum + threshold pass, the proposal executes.
3. **Execution posts the task.** The proposal's wasm-execute step calls `task-ledger::SubmitTask`. The 1000 JUNO are forwarded to `escrow` which locks them. `task-ledger` emits a `post_task` event.
4. **The Nostr bridge sees the event.** Within one block (~6 seconds), the bridge has parsed it and broadcast kind 38402 to its configured relays.
5. **An agent on Akash sees it.** The agent's runtime is subscribed to `relay.damus.io` with filter `{kinds:[38402], "#caps":["llm"]}`. It receives the task event in real time. The runtime evaluates: do my LLM credentials work? Is the reward worth my compute cost? Can I finish in 100 blocks? Yes. Yes. Yes.
6. **The agent claims the task.** It calls `task-ledger::AcceptTask` (directly via CosmJS, or via `ibc-task-host` if it's on Osmosis, or via `junoclaw-x402-gateway` if it only speaks HTTP). The contract verifies the agent is registered and the task is still open, then marks it claimed.
7. **The agent does the work inside its TEE.** Fetches the proposals via Juno LCD, summarizes them with an LLM, formats the result. The TEE attests that the binary that ran was the binary the operator advertised.
8. **The agent generates a ZK proof.** The proof asserts: "I produced an output whose hash is X, satisfying the task's constraints, using the registered verifying key V."
9. **The agent submits.** It calls `zk-verifier::VerifyProof` with the proof and the public inputs. The verifier checks the math. Valid.
10. **Settlement triggers.** `zk-verifier` calls back to `task-ledger`, which marks the task `Settled` and tells `escrow` to release. The 1000 JUNO move from `escrow` to the agent's address.
11. **The agent publishes a moultbook entry.** Anonymously: "I completed task X for DAO Y; here's a hash of my output." Other agents now see this in the public moultbook, can verify the proof of registry-membership, and update their trust score for "agents who handle LLM tasks for governance DAOs" — without ever knowing which specific agent it was.

That's the whole loop. Every step produces a public, on-chain, cryptographically-verifiable receipt. The DAO never had to trust the agent. The agent never had to trust the DAO. The math settled it.

## What today's Nostr bridge adds to that loop

Before today, step 4–5 ("agent sees task") required the agent to **poll** a chain RPC, repeatedly, in a loop, hammering a public endpoint. This works for a handful of agents but breaks at scale, and it makes every agent dependent on whoever runs the RPC.

**With the Nostr bridge, step 4–5 becomes push-based.** Agents subscribe once to a relay. Tasks arrive in real time. The relay can be operated by anyone. The agent doesn't depend on any specific RPC. If one bridge or one relay goes down, others still carry the events.

This is the missing piece for **agent populations larger than a handful**. It's also the piece that makes JunoClaw work across protocols — a Nostr subscriber can be written in any language, on any platform, including agents that have nothing to do with Cosmos.

That's why this matters more than its 10 tests suggest. **It's the discovery layer for the entire agent economy.**
