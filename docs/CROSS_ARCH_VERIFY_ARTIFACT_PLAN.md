# Cross-CPU/OS VerifySignature hash artifact — ARM sourcing + procedure

Last updated: 2026-06-23

## Goal

Prove that `hybrid.PubKey.VerifySignature` (Ed25519 + ML-DSA-44, via `cloudflare/circl`)
returns **bit-identical** results on x86_64 and on ARM/aarch64, and capture a small
signed artifact that records this. ML-DSA-44 is integer-only and deterministic, so the
two architectures MUST agree; this artifact is the evidence.

This is the last open item on the Phase F determinism checklist
(cross-impl verify is already done via the ML-DSA-44 KAT; only the cross-CPU/OS hash remains).

---

## Part A — Short-term ARM hardware (borrow or hire, NO ongoing commitment)

We only need ARM for a **single short run** (minutes). Pick the cheapest path that works.

### Option 1 — Borrow (free, preferred)

| Source | What to ask for | Notes |
|--------|-----------------|-------|
| **ffern** | "Can you run one Go test binary on your ARM box and send me a JSON file?" | He may already have Apple Silicon / a Pi. Zero cost. Self-contained: send the static binary + a one-line command. |
| **Any Apple Silicon Mac owner** | Same — run one binary, return one file | macOS/arm64 is a valid second platform (different OS *and* CPU = stronger evidence). |
| **A Raspberry Pi / Orange Pi owner** | Same | Linux/aarch64. |

Hand-off package (so the lender does zero setup):
- A statically built `verify-artifact` binary for `linux/arm64` (and/or `darwin/arm64`).
- One command: `./verify-artifact > arm.json`.
- They send back `arm.json`. Done.

### Option 2 — Hire/rent cloud ARM by the minute (no standing infra)

| Provider | Instance | Approx cost | Teardown |
|----------|----------|-------------|----------|
| **AWS** | `t4g.nano` (Graviton, aarch64) | ~$0.0042/hr (pennies) | `terminate-instances` immediately after |
| **Oracle Cloud** | `VM.Standard.A1.Flex` | Free tier eligible | Stop/terminate after run |
| **Hetzner** | `CAX11` (Ampere) | ~€0.005/hr | Delete server after run |
| **Scaleway** | `COPARM1-2C-8G` | per-minute billing | Delete after run |

Deterministic short-term recipe (AWS example, terminates same session):

```bash
# 1. launch (Ubuntu 24.04 arm64 AMI in your region)
AID=$(aws ec2 run-instances --image-id <ubuntu-arm64-ami> \
  --instance-type t4g.nano --key-name <key> \
  --query 'Instances[0].InstanceId' --output text)
# 2. wait, scp the binary, run it, scp back arm.json
# 3. ALWAYS terminate (no continuous cost):
aws ec2 terminate-instances --instance-ids "$AID"
```

> Rule: **terminate in the same session**. There is no recurring plan — one launch, one run,
> one terminate. Budget impact is sub-cent.

### Option 3 — Cross-build + QEMU emulation (no hardware at all)

Weakest evidence (emulated, not real silicon) but free and instant:

```bash
# build a linux/arm64 binary on x86_64
GOOS=linux GOARCH=arm64 go build -o verify-artifact-arm64 ./cmd/verify-artifact
# run under qemu-user
qemu-aarch64-static ./verify-artifact-arm64 > arm-qemu.json
```

Use Option 3 only as a stop-gap; prefer Option 1 or 2 for a real-silicon artifact.

## Part B — The artifact procedure (deterministic)

### B1. Fixed inputs (KAT)

Reuse the authoritative ML-DSA-44 vectors already in the repo
(`crypto/mldsa44/testdata/mldsa44_kat.json`, generated by the Rust `fips204` crate):
- fixed seed for keygen,
- fixed message,
- fixed signature.

Plus a fixed Ed25519 half (deterministic from a fixed seed) to form the hybrid key.

### B2. The tiny program (`cmd/verify-artifact`)

Pseudocode — load the KAT, run verify, hash the result deterministically:

```go
// load fixed msg, hybrid pubkey, hybrid signature from embedded KAT
ok := pub.VerifySignature(msg, sig)          // MUST be true
tampered := append([]byte{}, msg...); tampered[0] ^= 0x01
bad := pub.VerifySignature(tampered, sig)    // MUST be false

// deterministic artifact
h := sha256.Sum256(append(pub.Bytes(), append(sig, msg...)...))
out := map[string]any{
    "goos": runtime.GOOS, "goarch": runtime.GOARCH,
    "go_version": runtime.Version(),
    "verify_ok": ok, "tampered_rejected": !bad,
    "input_sha256": hex.EncodeToString(h[:]),
}
json.NewEncoder(os.Stdout).Encode(out)
```

The cross-arch invariant is: **`verify_ok`, `tampered_rejected`, and `input_sha256`
are identical on every platform**; only `goos`/`goarch`/`go_version` differ.

### B3. Run on both platforms

```bash
# x86_64 (this machine / WSL)
go run ./cmd/verify-artifact > artifacts/verify-amd64.json
# arm64 (Option 1/2/3 above)
./verify-artifact > artifacts/verify-arm64.json
```

### B4. Compare + store

```bash
jq -S 'del(.goos,.goarch,.go_version)' artifacts/verify-amd64.json > /tmp/a.json
jq -S 'del(.goos,.goarch,.go_version)' artifacts/verify-arm64.json > /tmp/b.json
diff /tmp/a.json /tmp/b.json && echo "CROSS-ARCH DETERMINISM CONFIRMED"
```

Deterministic outcome:
- `diff` is empty -> print `CROSS-ARCH DETERMINISM CONFIRMED`.
- Commit both JSONs to `aegis-bench/artifacts/` as the evidence artifact.
- If `diff` is non-empty -> a real determinism bug in the verify path; capture both files and investigate `circl` ML-DSA-44 on that platform.
