## Full context (key threads)

### 1. Structured-memory convention

Jake Hartnell's PR #929 contract code references `memory/v30-upgrade-plan.md B1+B2+B4` inline in the `BeforeValidatorSlashed` swallow comment. This is a **structured-memory citation pattern** — short markdown files in a `memory/` directory, referenced from code with section identifiers, used as the canonical recall layer for cross-cutting context.

**We adopted the same pattern this session.** The folder structure and INDEX format are documented in [`memory/INDEX.md`](./INDEX.md). The benefit is that future code comments in JunoClaw can reference `memory/<topic>.md` the same way, keeping the codebase searchable from any context-loading session.

### 2. LavaMoat

LavaMoat is the supply-chain hardening framework used by MetaMask. Mechanism: SES (Secure ECMAScript) sandboxing + allow-scripts policy + per-package compartments. Each npm package gets a constrained authority: its own ambient state, no global mutable access, no fs/network unless allowlisted.

**Relevance to JunoClaw.** Our agent-residue corpus pattern (where agents accumulate tool-use evidence over time) faces the same authority-constraint problem: how do we let agents do work without granting them ambient capability? LavaMoat is one answer; the other is **on-chain attestation gates** (which is what `task-ledger` + `zk-verifier` already do, in a sense). Worth a follow-up exploration: can we layer LavaMoat-style compartments on top of WAVS-attestation gates for in-process agent execution?
