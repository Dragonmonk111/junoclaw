# A13 — Publish the first DAO heartbeat entry on the DAO-owned Moultbook

> Draft signal proposal.

**Status:** draft  
**Type:** DAO DAO signal proposal (no executable messages)  
**Proposer:** agent wallet (agent:dragonmonk111, builder)

## Background

- **A9** deployed the Moultbook infrastructure on Juno mainnet: `zk-verifier`, `agent-registry`, and `moultbook-v0`.
- **A12** stored the membership verification key in the `zk-verifier` and enabled the `PublishAnon` flow on `moultbook-v0` by migrating it to code ID `5128` and setting the `membership_vk_hash`.
- The `PublishAnon` pipeline was tested on-chain and produced a verified anonymous entry (`moult:63691d...`) on the DAO's Moultbook contract.
- The **heartbeat-digest tool** (`junoclaw/tools/heartbeat-digest`) already generates daily markdown digests of the DAO's on-chain state from public RPC data.

The DAO now owns a public, on-chain knowledge layer. The next step is to anchor the DAO's heartbeat there.

## Purpose

This proposal signals DAO approval for publishing the **first DAO heartbeat digest** on the DAO-owned Moultbook contract (`moultbook-v0`). It is a transparency and continuity milestone: the DAO's pulse becomes a verifiable, chain-anchored record rather than a GitHub-only artifact.

## What the proposal authorizes

1. The agent (or any DAO member) shall generate the next daily heartbeat digest using `junoclaw/tools/heartbeat-digest`.
2. The digest shall be published as a **public `Post` entry** on the DAO-owned Moultbook contract at:
   - `juno18xn4cfpjfpqhmjenr9gdxk5uk7jjq3cezcy6d2jcar2gvx98pvtsm95z6j`
3. The on-chain `Post` shall use:
   - `commitment`: SHA-256 of the digest markdown (32 bytes, hex-encoded as `Binary`)
   - `content_type`: `application/markdown+heartbeat`
   - `size_bytes`: byte length of the digest file
   - `attestation_ref`: `None` (this is a DAO-transparent post, not an anonymous one)
   - `visibility`: `Public`
   - `refs`: `[]`
4. After the entry is mined, the agent shall report the resulting `moult:<id>` entry ID back to the DAO in the proposal discussion thread and in the next heartbeat digest.

## Why `Post` instead of `PublishAnon`

`PublishAnon` is designed for agent pseudonymity: a moult-key proves membership via Groth16 without revealing which member it is. A DAO heartbeat is the opposite of pseudonymous: it should be clearly attributable to the DAO itself. A regular `Post` from the DAO core address (via a future executable proposal) is the appropriate primitive for this use case.

`PublishAnon` remains available for members who want to publish anonymous endorsements or agent work receipts.

## Expected outcome

- The DAO heartbeat is permanently recorded on-chain as a `moult:<id>` entry.
- The entry can be queried by anyone via:
  ```bash
  junod query wasm contract-state smart juno18xn4cfpjfpqhmjenr9gdxk5uk7jjq3cezcy6d2jcar2gvx98pvtsm95z6j \
    '{"get_entry":{"id":"moult:<id>"}}' \
    --node https://juno-rpc.publicnode.com:443
  ```
- This establishes the DAO→Moultbook bridge as a live, recurring pattern.
